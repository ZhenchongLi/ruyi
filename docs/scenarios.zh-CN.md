## 引言

Ruyi（如意）是 Claude Code 的流程引擎。单独使用 Claude Code 时，你得到的是一个强大但需要人工盯着的 AI 助手。加上如意之后，你得到的是一个**自主运行的双代理系统**：

- **Agent A**（实现者）：Claude Code 全能力模式，读代码、写代码、跑测试
- **Agent B**（审查者）：独立的 Claude Code 实例，对抗性审查，找问题
- **如意**（裁判）：决定提交、修订还是回滚

三重保障让你放心交出复杂任务：

| 保障 | 说明 |
|------|------|
| **Worktree 隔离** | 每个任务在独立 git worktree 中运行，不碰你的工作目录 |
| **原子性提交/回滚** | 每个子任务要么全部提交，要么完整回滚，没有半成品 |
| **可恢复执行** | 中断后 `ruyi do` 自动从断点继续，已完成的子任务不会重跑 |

## 1. 大规模代码重构

### 适用情境

- 跨多个文件的 API 重命名（如将 `getUserInfo` 改为 `fetchUserProfile`）
- 框架升级迁移（如 React Class → Hooks、Express → Fastify）
- 架构调整（如将单体拆分为模块、提取公共库）
- 代码风格统一（如回调改 async/await、CommonJS 改 ESM）

这类任务改动面广、容易遗漏，人工做费时且容易引入 bug。如意的双代理审查能捕获遗漏的引用和不一致。

### 命令示例

**简单用法——一句话描述目标：**

```bash
ruyi do "将所有 API 路由从 Express 迁移到 Fastify，保持接口不变，更新对应的测试"
```

如意会与你交互确认计划、拆分子任务，然后自主执行。

**进阶用法——通过文件精确控制：**

创建 `refactor-plan.md`：

```markdown
将项目中所有 callback 风格的异步代码迁移到 async/await。

重点：
- src/services/ 下的所有文件
- src/controllers/ 下的所有文件
- 不要动 src/legacy/ 目录（第三方依赖）
- 每个文件的改动需要保证对应测试通过
```

然后执行：

```bash
ruyi do @refactor-plan.md
```

### 参数调优建议

在如意的交互规划阶段，你可以用自然语言调整参数：

- **"每个子任务改动不要超过 200 行"** → 如意会设置 `max-diff 200`，防止单次改动过大
- **"严格审查，分数要 9 分以上"** → `min-score 9`，审查者更严格
- **"不要动 config/ 和 migrations/ 目录"** → `forbidden` 列表保护关键文件
- **"最多改 3 轮"** → `max-revisions 3`，控制迭代次数

### 预期效果

- 如意自动创建 worktree，在隔离环境中逐个完成子任务
- 每个子任务完成后，Agent B 独立审查，不通过则自动修订
- 全部通过后自动创建 PR，包含所有子任务的提交
- 你的工作目录全程不受影响

### 注意事项

- 重构前确保项目有足够的测试覆盖，这样如意的 `build/test` 验证才有意义
- 对于超大范围的重构，建议在目标描述中明确优先级，如意会据此安排子任务顺序
- 如果中途中断（Ctrl+C 或网络问题），直接 `ruyi do` 重跑即可，已完成的不会重做

## 2. 代码审查与质量提升

### 适用情境

- 提升已有代码的质量（消除 code smell、统一错误处理）
- 安全审计（查找 SQL 注入、XSS、硬编码密钥等）
- 性能优化（找出 N+1 查询、不必要的重渲染、内存泄漏）
- 补充缺失的测试用例

### 命令示例

**代码质量提升：**

```bash
ruyi do "审查 src/api/ 目录，修复所有错误处理不一致的地方：统一使用 AppError 类，确保所有 async handler 都有 try-catch，移除裸的 console.log"
```

**安全审计：**

```bash
ruyi do "对 src/ 进行安全审查：检查 SQL 注入风险、XSS 漏洞、硬编码的密钥和 token，修复发现的问题并添加对应的测试"
```

**补充测试：**

```bash
ruyi do "为 src/services/ 下所有缺少单元测试的函数补充测试，覆盖正常路径和边界情况，测试文件放在 tests/services/ 对应目录"
```

### `judgement` 的高级用法

`judgement` 是给 Agent B（审查者）的评审标准。你可以在规划阶段通过自然语言设定：

- **"审查时重点关注线程安全和并发问题"** — 审查者会特别留意锁、竞态条件
- **"审查标准：每个函数不超过 30 行，圈复杂度不超过 10"** — 量化的代码质量标准
- **"用 OWASP Top 10 作为审查清单"** — 安全导向的审查

这比人工 code review 更一致——Agent B 每次都按同样的标准审查，不会因为疲劳而降低标准。

### 预期效果

- 如意拆分为若干子任务（按文件或按类别），逐个处理
- Agent A 修复问题，Agent B 验证修复是否到位、有无引入新问题
- 每一轮都在上一轮的基础上改进，质量逐步提升
- 最终产出是一个干净的 PR，包含所有改进和对应的测试

### 注意事项

- 设置合适的 `min-score`：质量提升类任务建议 8-9 分，安全审计建议 9-10 分
- 使用 `context` 指定相关的配置文件（如 ESLint 配置、安全策略文档），帮助 Agent 理解项目规范
- 纯审查不涉及新功能，建议在目标中明确"不要添加新功能，只修复现有问题"

## 3. 批量处理 GitHub Issue

### 适用情境

- 积压的 bug 需要批量清理
- 多个相关的 feature request 可以一起处理
- Sprint 规划中的一批小任务需要快速完成

### 命令示例

**处理单个 issue：**

```bash
ruyi do #42
```

如意会自动读取 issue 标题和描述，与你确认理解是否正确，然后自主实现。

**带额外说明的 issue：**

```bash
ruyi do #42 "用 Redis 而不是内存缓存来实现，参考 src/cache/redis.ts 的现有封装"
```

**批量并行处理多个 issue：**

```bash
ruyi pdo "#101" // "#102" // "#103"
```

每个 issue 在独立的 worktree 中并行处理，互不干扰，各自生成 PR。

### 工作流程

```
ruyi do #42
  │
  ├─ 读取 GitHub issue #42 的标题和内容
  ├─ 与你交互确认理解和实现方案
  ├─ 拆分为子任务（如有需要）
  ├─ Agent A 实现 → Agent B 审查 → 迭代
  └─ 创建 PR，自动关联 issue #42
```

### 预期效果

- 每个 issue 生成一个独立的 PR
- PR 描述中包含 issue 的关联信息
- 如果 issue 描述清晰，整个过程几乎不需要人工干预
- 使用 `pdo` 时，多个 issue 并行处理，速度成倍提升

### 注意事项

- issue 描述越清晰，如意的理解越准确。模糊的 issue 会触发更多的交互确认
- 并行处理时，确保 issue 之间没有代码冲突（不修改同一个文件）
- 复杂 issue 建议单独处理（`ruyi do #42`），简单 bug 适合批量并行（`ruyi pdo`）
- 如意会读取 issue 的 label 和评论，你可以在 issue 中预先留下实现提示

## 4. 文档与翻译任务

### 适用情境

- 将 README、文档翻译为多语言版本
- 批量更新 API 文档
- 编写技术博客或发布说明
- 生成 CHANGELOG 或迁移指南

### 命令示例

**翻译任务：**

```bash
ruyi do "将 README.md 翻译为日语版 README.ja.md，保持格式和链接不变，技术术语保留英文"
```

**批量翻译：**

```bash
ruyi pdo "将 docs/guide.md 翻译为中文 docs/guide.zh-CN.md" // "将 docs/api.md 翻译为中文 docs/api.zh-CN.md"
```

**文档撰写：**

```bash
ruyi do "根据最近 10 个 commit 的改动，为 v2.0 写一份发布说明，包含新功能、破坏性变更和迁移指南，中英双语"
```

**API 文档更新：**

```bash
ruyi do "根据 src/api/ 下所有路由的实际实现，更新 docs/api.md，确保每个端点的参数、返回值和示例与代码一致"
```

### 文档任务的特殊配置

文档任务通常不需要 build 和 test（如意会自动跳过），但 `judgement` 特别重要：

- **"翻译要自然流畅，不是逐句翻译，读起来像原生写的"** — 避免机翻感
- **"技术术语保留英文，但首次出现时附上中文解释"** — 兼顾专业性和可读性
- **"文档中的代码示例必须可运行"** — 审查者会检查示例的正确性

### 预期效果

- 翻译任务：生成格式完整的目标语言文档，链接和代码块保持不变
- 文档撰写：结构清晰、内容准确的技术文档
- Agent B 会从读者视角审查：信息是否完整、表述是否清晰、示例是否正确

### 注意事项

- 翻译大文档时，如意会自动拆分为多个子任务（按章节），每个子任务独立审查
- 使用 `context` 指定术语表或风格指南，保证翻译一致性
- 对于已有翻译的更新，明确说明"只更新变动部分，不要重新翻译整篇"

## 5. 团队级自动化配置

### 适用情境

- 团队希望统一使用如意处理特定类型的任务
- 需要建立标准化的代码质量流程
- 想让新成员快速上手 ruyi + Claude Code 的工作方式

### 在 CLAUDE.md 中配置

在项目根目录的 `CLAUDE.md` 中添加如意的指引，Claude Code 会自动遵循：

**基础配置：**

```markdown
# 开发规范

遇到以下类型的任务，使用 `ruyi do` 而不是直接实现：
- 涉及 3 个以上文件的改动
- 重构或迁移类任务
- 需要审查的代码质量改进

Ruyi 会处理 worktree 隔离、双代理审查和安全的提交/回滚。
```

**严格配置——适合核心模块：**

```markdown
# 核心模块开发规范

对 src/core/ 和 src/auth/ 的任何修改，必须使用 `ruyi do`：
- 审查分数至少 9 分
- 最多 3 轮修订
- 禁止修改 src/core/config.ts 和 src/auth/keys/
- 使用 opus 模型作为审查者
```

**轻量配置——适合文档和配置：**

```markdown
# 文档任务

文档和翻译类任务使用 `ruyi do`，审查分数 7 分即可。
重点关注可读性和准确性，不需要运行测试。
```

### 共享任务模板

团队可以将常用的任务定义为文件，提交到仓库：

```
tasks/
  security-audit.md    # 安全审计的标准目标描述
  release-notes.md     # 发布说明的模板
  i18n-review.md       # 国际化审查清单
```

团队成员直接引用：

```bash
ruyi do @tasks/security-audit.md
```

### 与 CI/CD 集成的思路

虽然如意目前是本地工具，但你可以在 CI 中触发：

```yaml
# .github/workflows/ruyi-audit.yml (概念示例)
on:
  schedule:
    - cron: '0 2 * * 1'  # 每周一凌晨 2 点
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run security audit
        run: ruyi do @tasks/security-audit.md
```

### 预期效果

- 团队成员在 Claude Code 中工作时，自动被引导使用如意处理复杂任务
- 统一的审查标准确保代码质量一致
- 新成员通过 `CLAUDE.md` 和任务模板快速了解团队规范

### 注意事项

- `CLAUDE.md` 中的规则是给 Claude Code 看的指引，不是强制约束
- 任务模板要保持更新，过时的模板比没有模板更糟
- 建议在团队 wiki 中记录"什么时候用 ruyi，什么时候直接让 Claude Code 做"的判断标准

## 6. 并行执行多任务

### 适用情境

- 多个独立的任务需要同时进行（如多个不相关的 bug fix）
- 多语言翻译（中文、日文、韩文同时进行）
- 多个模块的独立改进
- 需要快速清理一批积压任务

### 命令示例

**基本并行：**

```bash
ruyi pdo "修复登录页的 XSS 漏洞" // "为支付模块添加单元测试" // "更新 API 文档"
```

三个任务在三个独立的 worktree 中同时执行，各自有自己的 Agent A 和 Agent B。

**并行处理 issue：**

```bash
ruyi pdo "#11" // "#15" // "#23"
```

**并行翻译：**

```bash
ruyi pdo "翻译 README 为日语" // "翻译 README 为韩语" // "翻译 README 为法语"
```

### 工作原理

```
ruyi pdo "任务1" // "任务2" // "任务3"
  │
  ├─ Worktree 1: 任务1 → Agent A₁ 实现 → Agent B₁ 审查 → PR₁
  ├─ Worktree 2: 任务2 → Agent A₂ 实现 → Agent B₂ 审查 → PR₂
  └─ Worktree 3: 任务3 → Agent A₃ 实现 → Agent B₃ 审查 → PR₃
```

每个任务完全独立：独立的 worktree、独立的 Agent 对、独立的 PR。

### 预期效果

- 三个任务同时开始，总耗时约等于最慢的那个任务
- 每个任务生成独立的 PR，方便分别审查和合并
- 终端中可以看到所有任务的实时进度

### 注意事项

- **确保任务之间无代码冲突**：并行任务修改同一个文件会导致合并冲突。如果有冲突风险，用 `ruyi do` 顺序执行
- **资源消耗**：每个并行任务都会启动一个 Claude Code 实例，注意 API 额度。3-5 个并行任务是比较合理的范围
- **规划阶段是顺序的**：`pdo` 会依次与你确认每个任务的计划，全部确认后才并行执行
- 如果某个任务失败，不影响其他任务的继续执行

## 小结

| 场景 | 命令 | 关键参数 |
|------|------|----------|
| 重构 | `ruyi do "重构目标"` | `max-diff`, `forbidden` |
| 审查 | `ruyi do "审查目标"` | `min-score 9`, `judgement` |
| Issue | `ruyi do #123` | — |
| 翻译 | `ruyi do "翻译目标"` | `judgement`（翻译标准） |
| 团队 | 配置 `CLAUDE.md` | — |
| 并行 | `ruyi pdo "A" // "B"` | 确保无冲突 |

记住如意的核心价值：**你定义"做好"的标准，如意确保达到标准**。命令越简单越好，细节交给交互规划阶段。
